# Output Specifications

Defines the canonical output contract for DiceLab runs and clarifies how benchmark timing data is reported.

This file controls program output shape.
Benchmark methodology and timing procedure are defined in `benchmark_spec.md`.

## 1. Scope

Two output categories are defined:

1. **Simulation Output (required)**
   - Produced by normal CLI runs (`text`, `json`, `csv`)
   - Contains simulation results and statistical summary

2. **Benchmark Report Output (optional, separate artifact)**
   - Produced by benchmark tooling/scripts
   - Contains timing/performance measurements across repeated runs

Timing fields must not be mixed into baseline simulation output formats.

## 2. Required Simulation Fields

All simulation output formats must represent the same data:

- `total_rolls`
- `sides`
- `distribution` (face → count)
- `mean`
- `variance`
- `std_dev`

## 3. Ordering Rules

- Distribution rows/keys must be ordered by face ascending (`1, 2, ..., n`).
- Output ordering must be deterministic for identical inputs within the same implementation.

## 4. Text Output Format (`--format text`)

Required sections:

- Header
- Total rolls
- Sides
- Distribution table
- Summary statistics

Distribution row format:

`face | count | percentage`

Percentage should be computed as `(count / total_rolls) * 100`.

## 5. JSON Output Format (`--format json`)

Required structure:

```json
{
  "total_rolls": 10000,
  "sides": 6,
  "distribution": {
    "1": 1660,
    "2": 1704
  },
  "mean": 3.51,
  "variance": 2.93,
  "std_dev": 1.71
}
```

Notes:

- Distribution keys are face values serialized as strings.
- Keys must be emitted in ascending numeric face order.

## 6. CSV Output Format (`--format csv`)

CSV must include a distribution header row:

`face,count,percentage`

Each subsequent row represents one face in ascending order.

If summary metrics are included in CSV, they must be placed in a clearly separated summary block.

## 7. Benchmark Timing and Performance Reporting

Timing/performance values are required for benchmark publication, but they belong to benchmark output artifacts, not baseline simulation output.

Required benchmark metrics are defined by `benchmark_spec.md`, including:

- Mean execution time
- Standard deviation of execution time (if measured)
- Optional min/max

### 7.1 Recommended Benchmark Report JSON

```json
{
  "language": "python",
  "implementation": "baseline",
  "workload": {
    "rolls": 10000,
    "sides": 6,
    "parallel": false
  },
  "timing_ms": {
    "runs": [12.3, 12.0, 12.5, 12.1, 12.2],
    "mean": 12.22,
    "std_dev": 0.17,
    "min": 12.0,
    "max": 12.5
  },
  "environment": {
    "os": "...",
    "cpu": "...",
    "cores": 8,
    "ram_gb": 32,
    "language_version": "..."
  }
}
```

This report is generated by benchmark tooling and should not replace normal CLI simulation output.

## 8. Normalization Constraints (Benchmark Safety)

To avoid polluting timing measurements:

- No intermediate roll logging
- No debug output during benchmark runs
- Final output only
- No nondeterministic ordering artifacts

## 9. Versioning Guidance

If output schema changes:

- Update this spec first
- Update all implementations consistently
- Document changes in benchmark/article write-ups
