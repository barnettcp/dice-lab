DiceLab Benchmark Report

DiceLab cross-language benchmark — C++, Go, Java, Python, Rust

1 — Introduction

This report presents the results of the DiceLab benchmark, a cross-language experiment measuring the execution time of a simulated dice-rolling workload implemented identically in five languages: C++, Go, Java, Python, and Rust.

The experiment is designed to answer three questions:

  • How do the five implementations compare at the same workload size?
  • How does each implementation scale as the number of rolls grows from 100 to 1 million?
  • How consistent is timing within each language across repeated runs?

Methodology

Each benchmark session consists of 10 batch runs. Within each batch, every language is executed at five workload levels (100, 1 K, 10 K, 100 K, and 1 M rolls), each repeated for 5 trials. This yields 50 timed observations per (language, workload) pair — 1 250 rows in total.

All implementations roll a configurable number of six-sided dice using a pseudo-random number generator seeded per run. Timing captures only the roll loop itself, excluding process start-up and I/O overhead as much as the CLI wrapper permits.

2 — Batch Timing Overview

Before examining individual languages, it is useful to look at how long each complete benchmark batch took end-to-end. Each batch exercises all five languages across all five workload levels for five trials — so a single batch applies 125 timed executions.

The chart below plots total batch elapsed time (in milliseconds) against batch run index. A downward trend at the start is the expected signature of warm-up effects: the operating system's branch predictor, file-system cache, and — in Java's case — the JVM's JIT compiler all improve over the first few runs. A rising trend or high late-run variance would suggest sustained system load or thermal throttling.

The first batch completed in 7.53 s; the last in 6.87 s. The mean across all ten batches was 7.03 s.

Use the dropdown to inspect magnitude comparisons at a single fixed workload. Error bars show ±1 standard deviation across the 50 trials. This view is particularly effective for seeing how dominant Java's JVM startup overhead is at small workloads, and whether that gap closes at larger ones.

Timing Consistency (Coefficient of Variation)

Raw standard deviation cannot be fairly compared between a language timing at ~12 ms (Rust) and one at ~65 ms (Python) — the absolute spread naturally differs. The coefficient of variation (CV = std / mean × 100 %) normalises spread relative to the mean, making consistency directly comparable across all languages.

A flat, low CV indicates reliable timing. A high or climbing CV suggests the language is susceptible to OS scheduling jitter, JIT warm-up variability, or memory-pressure effects at that workload level.
